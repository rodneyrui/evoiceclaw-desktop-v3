一、逻辑层总体框架
1.1 核心原则
安全第一：任何功能必须在可控范围内执行，越权操作必须被阻止并记录。

用户主权：用户对自己的数据和工作区拥有完全控制权，系统不得擅自跨越用户设定的边界。

最小权限：每个组件（Skill、工具、Agent）只拥有完成任务所需的最小权限，且权限必须在用户知情下授予。

可追溯：所有敏感操作必须记录审计日志，便于事后审查和自我改进。

自举有限性：AI可以辅助自身改进，但关键决策（如安全边界调整）必须由用户或可信机制确认。

1.2 分层决策权
层级	决策主体	决策范围
L0：用户	最终用户	全局设置、工作区创建、敏感操作授权
L1：系统内核	内置逻辑 + 守门员	工具调用权限、Skill安装审查、安全策略执行
L2：Agent/Skill	LLM + 规则	在授权范围内选择工具、生成回复
L3：自举辅助	AI（如Opus/R1）	提供建议、生成代码、审查结果，但无执行权
二、子系统逻辑规则
2.1 工作区隔离规则
规则项	逻辑定义
文件系统隔离	每个工作区拥有独立的根目录（如 workspaces/{id}/），所有文件操作工具必须通过路径校验，确保访问路径在该目录下。禁止使用 .. 或绝对路径逃逸。
记忆系统隔离	LanceDB 中所有表按 workspace_id 分区，查询和写入时必须携带当前工作区ID，禁止跨区访问。
网络权限隔离	每个工作区可独立配置域名白名单，NetworkGuard 根据工作区ID加载对应规则。
环境变量隔离	每个工作区可设置独立的环境变量（如 API Key），但需加密存储，且只在该工作区进程内可见。
跨区通信	默认禁止。如有需要，必须通过系统提供的“事件桥接”机制，且需用户显式授权（如：允许工作区A向工作区B发布事件）。
2.2 Shell 工具逻辑规则
2.2.1 启用条件
默认状态：全局禁用。

启用方式：用户必须在工作区设置中显式开启“允许Shell命令”，并选择安全级别（低/中/高）。

低：仅允许预定义安全命令（如 date、echo），禁止管道、重定向、后台进程。

中：允许指定命令列表（如 himalaya、python3 -c），需守门员审查命令合理性。

高：允许任意命令，但必须经过三层沙箱 + 用户实时审批（每命令弹窗确认）。

2.2.2 执行流程
text
Skill请求执行命令 → ① 检查工作区是否启用Shell → ② 匹配安全级别规则 → ③ 命令预处理（参数检查） → ④ 运行时沙箱（超时、目录限制） → ⑤ 执行并记录审计 → ⑥ 返回结果
2.2.3 安全规则
命令白名单：由系统维护一个基础安全命令列表（如 ls、cat、grep 等无副作用命令），用户可扩展但需守门员审查。

参数过滤：禁止包含 ;、&&、||、|、>、<、$()、` 等特殊字符，除非用户级别为“高”且实时审批通过。

资源限制：

超时：默认30秒，可配置

输出大小：默认1MB，防止无限输出

进程数：禁止 fork 炸弹（通过 ulimit -u 限制）

环境清理：执行时仅传递最小环境变量（PATH、HOME 等），不传递用户敏感变量。

2.2.4 审计要求
每条命令的执行记录：时间、工作区ID、Skill名称、命令全文、执行结果（成功/失败）、输出摘要。

审计日志不可篡改（只追加），定期备份。

2.3 代码审核工具（review_code）逻辑规则
2.3.1 审核范围
文件类型：仅审核工作区内的源代码文件（可配置扩展名，如 .py、.js、.md）。

审核粒度：

基础：语法错误、潜在空指针、未定义变量

安全：SQL注入、XSS、路径遍历、硬编码密钥

性能：循环嵌套过深、内存泄漏风险

风格：命名规范、注释密度、复杂度（可配置规则集）

2.3.2 审核流程
text
用户请求审核 → ① 读取指定文件/diff → ② 调用高能力LLM（如路由到最强模型） → ③ LLM按规则输出结构化结果 → ④ 结果存入LanceDB（供后续学习） → ⑤ 返回给用户
2.3.3 权限限制
只能读取当前工作区内的文件，不能跨区。

审核结果可写入工作区，但不能修改原文件（修改需通过 edit_skill 等工具）。

审核过程中调用的LLM必须是用户信任的（通过模型矩阵配置），且调用日志记录。

2.3.4 自举标记
审核结果中如果包含修改建议，且用户采纳后提交代码，应在 commit message 中自动添加 [AI reviewed] 标记。

审核记录本身也可作为“AI辅助开发”的证据，可在项目 /meta 目录中留存。

2.4 Skill 安装与修改规则（守门员 Gatekeeper）
2.4.1 安装前审查
守门员必须解析 SKILL.md，提取：

所需工具列表

网络域名声明

命令模式声明（如 Shell 命令）

潜在恶意模式（如尝试读取敏感文件）

守门员根据系统安全策略进行评分，若风险过高则拒绝安装；若中等风险则改写为安全版本（如限制命令参数）。

审查结果生成 JSON 报告，与 Skill 一同存储。

2.4.2 安装时权限声明
安装前向用户展示：

此 Skill 将获得以下权限：

读取工作区内的 .txt 文件

访问域名 api.weather.com

执行命令：python3 -c "..."（仅限单行）

是否继续？

用户可选择“允许一次”、“始终允许（此工作区）”、“拒绝”。

2.4.3 修改 Skill（edit_skill）
仅限“开发者模式”工作区（需用户主动开启）。

修改后必须重新触发守门员审查，若未通过则自动回滚并提示。

每次修改记录审计：修改者、时间、变更内容摘要。

2.5 模型矩阵与路由规则
2.5.1 模型选择逻辑
系统维护一个能力评分矩阵（你已有的14维）。

路由规则：

简单任务（如天气查询） → 低成本模型

中等任务（如代码生成） → 中等能力模型

复杂任务（如推理、安全审查） → 最高能力模型

成本超限时自动降级（用户可配置预算上限）

2.5.2 模型健康管理
每个 Provider 有健康状态（正常/降级/离线），基于响应时间、错误率动态调整。

失败时自动重试（指数退避），超过阈值切换备用模型。

2.5.3 隐私保护
发送到云端模型的文本必须经过隐私管道脱敏（如替换敏感实体为占位符）。

用户可配置“禁用云端模型”，仅使用本地模型（功能可能受限，但数据不出设备）。

2.6 隐私管道逻辑
阶段	逻辑规则
① 认知隔离	识别敏感信息（正则+语义），分离出敏感流（加密存储）和清洁流（继续处理）。敏感流中的实体替换为占位符（如 [NAME_1]）。
② 实体映射	清洁流中的实体通过向量检索匹配已知类型，丰富上下文。匹配结果存入 LanceDB。
③ 上下文压缩	按重要性评分压缩历史对话，保留最近 N 轮 + 关键实体相关记忆。评分规则：时间衰减 + 实体关联度 + 用户显式标记。
④ 记忆注入	从 LanceDB 检索与当前查询相关的记忆（向量相似度 + 元过滤），按优先级注入 prompt。
⑤ 记忆蒸馏	会话结束后，抽取新事实/偏好，向量化后 merge 到 LanceDB。冲突解决：新覆盖旧（按实体ID+类型）。
2.7 自举与证据留存规则
2.7.1 可自举的范围
允许 AI 辅助完成：代码生成、文档撰写、审核、测试用例生成、社区回复。

禁止 AI 自行修改：安全策略配置文件、用户密钥存储、审计日志。

2.7.2 标记规范
所有 AI 辅助生成的内容（代码、文档、审核意见）应在对应位置留下标记：

代码：# AI-generated 或 // AI-assisted

文档：> 本节由 AI 辅助生成

审核意见：评论中注明 🤖 AI Code Reviewer

提交时，commit message 可包含 [AI] 标签。

2.7.3 公开透明度
项目 /meta/ai-traces/ 目录下可存放 AI 辅助记录（如审核日志、生成草稿），供社区查阅。

定期发布“自举报告”，统计 AI 辅助的代码行数、发现的 bug 数量、节省的时间等。

三、异常处理规则
异常场景	处理逻辑
Skill 试图越权	工具执行层拦截，返回错误，记录审计，并通知用户（可选：暂时禁用该 Skill）。
守门员不确定	若风险评分在阈值附近，暂停安装，请求用户人工判断，并记录该案例供后续优化。
模型超时/失败	自动降级到备用模型，若都失败则返回友好错误，提示用户稍后重试。
隐私管道脱敏失败	若无法安全脱敏（如识别出新型敏感信息），则拒绝处理该请求，并记录日志供改进。
用户操作冲突	如同时修改同一文件，采用“最后保存者胜”规则，但提醒用户冲突。
四、用户交互规则
4.1 权限请求
任何需要用户授权的操作（如安装 Skill、执行高风险命令），必须在前端弹窗，明确说明：

请求方（哪个 Skill/Agent）

请求的权限（读文件、写文件、网络、命令等）

持续时间（仅一次、永久、本次会话）

风险提示（如“此命令可删除文件，请确认”）

4.2 设置面板
工作区设置中提供：

安全级别（低/中/高）滑块

可自定义的命令白名单

可自定义的域名白名单

隐私管道开关（可关闭某些阶段，但会降低隐私保护）

审计日志查看入口

4.3 透明度报告
系统可生成“每周安全摘要”：显示本周执行的敏感操作数量、被拦截的越权尝试、AI 审核统计等，让用户了解系统运行状况。

五、逻辑层与AI的协作方式
当你想让 Opus 或 R1 协助实现某个功能时，不再问“可不可以”，而是提供以下信息：

我们有以下逻辑规则：

[规则1] ...

[规则2] ...

[规则3] ...

基于这些规则，请设计一个 [功能名] 的实现方案，包括：

需要新增哪些工具或接口？

在代码层面如何检查这些规则？

需要记录哪些审计信息？

用户交互流程应该怎样？

这样，AI 就可以在规则框架内自由发挥，而不必担心突破安全边界。
