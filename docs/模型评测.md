2.2 评测调度器（Evaluation Scheduler）
触发方式：

手动触发（用户添加新模型后）。

定时触发（每周/每月自动评测所有活跃模型）。

事件触发（模型供应商更新模型版本时，通过 API 通知）。

执行流程：

从配置中获取待评测模型列表（包括 API 密钥、端点等信息）。
对每个模型，依次调用 55 道题，记录：
模型回复内容
响应时间
Token 消耗
对回复内容进行自动评分（需一个评分模型，见 2.3）。
汇总 11 维度得分，结合速度、Token、价格，生成 14 维矩阵。
将结果存入 LanceDB（model_evaluations 表）。
2.3 自动评分机制
这是核心难点——谁来评判模型的回答质量？有三种方案：

方案	原理	优点	缺点
A：规则匹配	对客观题（如数学计算）用正则或精确匹配；对主观题用关键词评分	速度快，成本低	无法处理开放性回答，准确率有限
B：专用评分模型	训练一个小型模型，输入问题和模型回答，输出分数	准确率高，可定制	需要训练数据，开发成本高
C：深度思考模型	调用一个高能力模型（如 GPT-4、Kimi K2.5）作为“裁判”，对回答评分	准确率最高，无需训练	成本高，速度慢
建议组合使用：

客观题（有确定答案）用方案 A。

主观题用方案 C，但可以缓存评分结果（同一题目对多个模型评分时，只需一次裁判调用）。

随着数据积累，逐步训练一个专用评分模型（方案 B），降低对高成本模型的依赖。

2.4 评测结果存储

timestamp	timestamp	评测时间
dimension_scores	json	11 维度得分，如 {"logic":0.95, "coding":0.88, ...}
avg_response_time	float	平均响应时间（秒）

cost_performance	float	性价比得分（可自定义公式）
raw_data	json	原始问答记录（可选，用于审计）

评测调度器状态机与模型能力矩阵更新事件机制
一、评测调度器（Evaluation Scheduler）设计
评测调度器负责管理所有模型的评测任务，包括触发、执行、状态跟踪和结果处理。它独立运行于系统后台，不阻塞用户正常使用。

1.1 评测任务生命周期状态机
每个模型对应一个评测任务，状态转换如下：
















状态说明：

PENDING：新模型被发现（如用户添加 API Key 后 list_models 返回的模型），但尚未加入评测队列。此时系统使用预置数据或默认值。

QUEUED：任务已加入等待队列，但尚未开始执行。触发条件包括：

用户手动点击“立即评测”。

系统空闲（CPU/GPU 使用率低于阈值，持续 N 分钟）。

定时任务（如每天凌晨 2 点）。

模型被首次使用（可选，可配置）。

RUNNING：正在执行评测题目（11 维度 × 5 题，共 55 次调用）。

COMPLETED：评测成功完成，结果已写入能力矩阵。

FAILED：评测过程中出现错误（网络错误、模型返回异常、超时等）。可配置重试次数。

CANCELLED：用户手动取消任务。

1.2 评测队列管理
队列优先级：可设置“用户手动触发” > “首次使用触发” > “空闲触发” > “定时触发”。同一优先级内按 FIFO。

并发限制：默认同时运行 1 个评测任务（避免耗尽 API 额度或系统资源），可配置。

存储：任务状态存储在 SQLite 表 evaluation_queue 中，字段包括：

task_id (str, 主键)

model_id (str, 如 openai/gpt-4)

status (enum: PENDING, QUEUED, RUNNING, COMPLETED, FAILED, CANCELLED)

priority (int, 0-255, 越小优先级越高)

retry_count (int)

created_at (timestamp)

started_at (timestamp, 可为空)

completed_at (timestamp, 可为空)

error_message (text, 可为空)

result_id (str, 关联到 model_evaluations 表)

1.3 空闲检测机制
空闲检测器定期（如每分钟）检查系统负载：

CPU 使用率：低于 20% 持续 5 分钟。

内存使用：可用内存 > 2GB。

用户活动：无键盘/鼠标输入超过 10 分钟（仅桌面环境，可通过系统 API 检测）。

网络：连通性正常（避免在断网时评测）。

满足条件时，从队列中取出下一个 QUEUED 任务开始执行。

1.4 评测执行器
从 evaluation_questions 加载 55 道题目（可配置是否随机顺序）。

依次调用模型 API，记录每次的：

响应内容

响应时间

token 消耗

是否成功

调用自动评分模块（规则+深度思考模型）给每个答案打分。

计算各维度平均分、总 token 消耗、总耗时、价格（根据模型单价）。

组装成 14 维矩阵数据。

写入 LanceDB model_evaluations 表。

更新任务状态为 COMPLETED，记录 result_id。

1.5 错误处理与重试
临时错误（如网络超时、5xx）可重试，最多 3 次，每次等待时间指数退避（2^n 分钟）。

永久错误（如认证失败、模型不存在）标记为 FAILED，不重试，并记录错误信息供用户查看。

部分失败处理：如果某道题失败，可跳过（记录失败原因），但整体任务仍可完成（得分基于成功题目）。

2.4 规则生成器的触发策略
为了避免频繁触发规则生成（可能消耗大模型资源），建议采用以下策略：

延迟合并：收到事件后，启动一个定时器（如 30 秒），等待更多评测完成事件，然后批量处理。

按需生成：规则生成器本身也是一个后台任务，可定期（如每天）根据最新评测数据生成规则，而不是每次评测都生成。

用户控制：在设置中提供“立即应用最新评测”按钮，手动触发生成。
