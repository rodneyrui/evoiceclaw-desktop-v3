AI OS 逻辑层宪法（v1.0）
序言
本宪法定义 eVoiceClaw AI OS 的内在运行法则。AI OS 视 LLM 为中央处理器（CPU），Skill 为应用程序，Function Calling 为指令集，记忆系统为持久化存储，隐私管道为内存管理单元，安全层为硬件级防护。所有组件——包括内核、工具、Skill、用户界面及辅助 AI（如 Opus、R1）——必须在本宪法框架内行动。

宪法目标：

安全：任何操作不得突破工作区隔离与用户主权。

自举：系统可借助自身能力持续进化，但关键决策必须可追溯、可审计。

透明：所有规则的执行结果对用户开放，供信任建立与社区审查。

第一编：总体原则
第1条 主权分层
用户层：拥有最高权限，可创建/销毁工作区、授予/撤销全局权限、查看审计日志。

内核层（AI OS 核心）：负责执行用户指令、调度资源、实施安全策略，无权篡改用户数据或越权操作。

应用层（Skill、Agent）：在用户授权的工作区内运行，只能通过内核提供的工具访问资源。

第2条 最小权限
任何模块（包括内核）仅拥有完成其功能所必需的最小权限，且权限必须在用户知情下授予。

权限不能自我提升——Skill 不能通过任何手段获得超出安装时声明的权限。

第3条 隔离即安全
工作区是基本隔离单元：文件系统、记忆、网络配置、环境变量均按工作区隔离。

工作区之间默认不可通信；跨区通信必须通过内核桥接，且需用户显式批准。

第4条 可追溯性
所有敏感操作（文件写、网络请求、Shell 执行、权限变更）必须记录审计日志。

审计日志不可篡改（仅追加），并可由用户导出。

第5条 自举有限性
AI 可以辅助生成代码、审查 Skill、优化配置，但无权修改安全策略、内核代码或用户密钥。

任何 AI 辅助的修改必须留下可识别的标记（如 [AI] 标签）。

第二编：内核资源管理
第6条 工作区管理器
工作区是资源容器，包含：

独立文件目录（workspaces/{id}/）

独立记忆分区（LanceDB 中所有表按 workspace_id 分片）

独立网络白名单

独立环境变量（加密存储）

内核保证任何工具调用无法访问当前工作区之外的路径、记忆或配置。

第7条 工具注册与调用
所有工具必须在内核的 ToolRegistry 中注册，声明：

名称、描述、参数模式（JSON Schema）

所需权限（读文件、写文件、网络、Shell 等）

执行前回调（用于权限检查）

工具调用时，内核依次执行：

权限检查：调用方（Skill/Agent）是否拥有该工具的使用权。
参数校验：是否符合 Schema。
执行：在受控环境中运行，并计时、计资源。
审计记录：将调用结果写入审计日志。
第8条 网络守卫（NetworkGuard）
每个工作区维护一个域名白名单。

网络工具（http_get、http_post、fetch_webpage）只能访问白名单内的域名。

默认白名单为空；Skill 安装时可声明所需域名，守门员审查合理性后，用户确认方可加入。

禁止访问内网 IP、本地端口及常见云元数据服务（如 169.254.169.254）。

第9条 记忆系统规则
记忆存储采用 LanceDB，按工作区分表。

提供三个核心工具：

recall_memory(query, top_k=5)：语义检索当前工作区记忆。

save_memory(content, metadata)：存储新记忆，自动向量化。

search_entities(text)：识别文本中的实体并匹配已知类型。

记忆蒸馏由内核定期触发：从会话日志中抽取新事实，合并到记忆库，并清理过期记忆。

第三编：Shell 执行宪章
第10条 Shell 的定位
Shell 是 AI OS 与底层操作系统之间的桥梁，允许 Skill 执行命令行程序。因其高敏感性，Shell 受最严格的约束。

第11条 启用条件
Shell 功能默认 全局禁用。用户必须在工作区设置中主动开启，并选择安全级别：

级别	命令范围	审批方式	适用场景
L1 - 沙箱模式	仅预定义安全命令（date, echo, ls, cat 等）	自动执行	基础信息查询
L2 - 受限模式	守门员审查过的命令列表（如 himalaya, python3 -c 单行）	自动执行	常用工具自动化
L3 - 高风险模式	任意命令	每条命令需用户实时弹窗确认	开发调试、紧急运维
第12条 命令执行流程
拦截：exec_command 被调用时，内核检查当前工作区的 Shell 级别。

静态过滤：

若为 L1：仅允许白名单命令，且参数禁止包含特殊字符（;, &&, |, > 等）。

若为 L2：命令必须在 Skill 安装时声明的模式内（如 python3 -c "..." 且内容受限）。

若为 L3：跳过静态过滤，进入用户审批。

动态沙箱：

超时：30 秒（可配置）

工作目录：锁定为工作区目录

输出限制：1 MB

进程数：禁止 fork（通过 ulimit -u 20）

环境变量：仅传递最小集（PATH=/usr/bin:/bin, HOME 等）

审计记录：无论成功与否，记录完整命令行、执行时间、退出码、输出摘要。

返回结果：截断输出后返回给调用方。

第13条 禁止令
无论何种级别，以下命令/操作永远禁止：

删除/修改系统文件（rm -rf /*, chmod 系统目录）

提权命令（sudo, su, chown）

网络穿透（ssh, scp, rsync 到外部）

直接操作硬件（dd, fdisk）

关闭审计日志或修改系统时间

第四编：Skill 生命周期管理
第14条 守门员（Gatekeeper）职责
守门员是一个专用 LLM（或规则引擎），负责 Skill 安装时的安全审查。其职责包括：

解析 SKILL.md，提取工具需求、网络域名、命令模式。

检测恶意模式（如尝试读取 /etc/passwd、隐藏网络回调）。

对可疑 Skill 进行改写（例如将危险命令替换为安全版本）。

生成审查报告（REVIEW.json），与 Skill 一同存储。

第15条 安装流程
用户请求安装 Skill（从本地或市场）。

守门员执行审查，输出改写后的 SKILL.md 及权限声明。

权限声明展示：

text
此 Skill 请求以下权限：
- 读取工作区内的 .txt 文件
- 写入工作区内的 output/ 目录
- 访问域名 api.openweathermap.org
- 执行命令：python3 -c "..."（L2 级别）
用户确认：可选择“允许一次”、“始终允许（此工作区）”、“拒绝”。

安装完成：Skill 文件存入 skills/{name}/，权限记录写入工作区配置。

第16条 Skill 修改（edit_skill）
edit_skill 工具仅限“开发者模式”工作区使用。

修改后必须重新触发守门员审查；若未通过，自动回滚并提示。

每次修改记录审计：修改者、时间、变更 diff。

第17条 Skill 市场规则
市场中的 Skill 必须经过守门员审查（公开审查报告）。

用户可对 Skill 评分、举报恶意行为。

内核定期拉取市场更新，但不自动安装。

第五编：隐私管道（数据流宪法）
第18条 数据流强制路径
所有进入 LLM 的用户数据必须经过隐私管道，顺序不可跳跃：

text
用户输入 → ①认知隔离 → ②实体映射 → ③上下文压缩 → ④记忆注入 → ⑤LLM 内核
第19条 各阶段规则
① 认知隔离：

使用正则 + 小模型识别敏感信息（身份证、银行卡、密码等）。

极高敏感数据替换为占位符（如 [REDACTED_ID]），并加密存储映射表。

输出两条流：敏感流（本地加密）、清洁流（继续管道）。

② 实体映射：

清洁流中的实体通过向量检索匹配已知类型（PERSON、ORG、LOC 等）。

匹配结果注入上下文，同时更新 LanceDB entities 表。

③ 上下文压缩：

按重要性评分压缩历史对话，保留最近 N 轮 + 关键实体相关记忆。

重要性评分公式：时间衰减系数 * (实体匹配分 + 用户显式标记分)。

④ 记忆注入：

从 LanceDB 检索与当前查询语义相关的记忆（向量相似度）。

按优先级（核心事实 > 近期记忆 > 蒸馏规则）注入 prompt。

⑤ 记忆蒸馏：

会话结束后，调用蒸馏 LLM 抽取新事实、偏好、规则。

向量化后合并到 LanceDB，冲突时新覆盖旧。

第20条 隐私恢复
LLM 回复中可能包含占位符（如 [NAME_1]），在返回用户前，内核根据加密映射表还原为原始信息。

若还原失败（如映射丢失），则保留占位符并提示用户。

第六编：模型矩阵与路由
第21条 模型即 CPU
系统维护一个模型能力评分矩阵（14 维），包括推理能力、成本、速度、上下文长度等。

路由规则由 SmartRouter 执行：

简单查询 → 低成本模型（如 DeepSeek Lite）

代码生成 → 中等能力模型（如 Claude Haiku）

安全审查/复杂推理 → 最高能力模型（如 GPT-4o / Kimi K2.5）

成本超限时自动降级（用户可配置月度预算）。

第22条 模型健康管理
每个 Provider 有健康状态（正常/降级/离线），基于响应时间、错误率动态调整。

调用失败时自动重试（指数退避，最多 3 次），超过阈值切换备用模型。

第23条 隐私分级
用户可配置“仅本地模型”模式，所有请求使用本地部署模型（功能可能受限，但数据不出设备）。

默认模式允许使用云端模型，但必须经过隐私管道脱敏。

第七编：自举与证据留存
第24条 可自举范围
允许 AI 辅助完成：

代码生成与审查（review_code）

文档撰写

测试用例生成

社区回复

Skill 推荐

禁止 AI 自行修改：

安全策略配置文件（security.yaml）

用户密钥存储

审计日志

内核代码（除非通过用户批准的 PR 流程）

第25条 自举标记
所有 AI 辅助生成的内容必须在对应位置留下标记：

代码注释：# AI-generated 或 // AI-assisted

文档：> 本节由 AI 辅助生成

审核意见：评论中注明 🤖 AI Code Reviewer

Commit message：包含 [AI] 标签

内核提供工具 add_ai_marker() 自动添加标记。

第26条 透明度
项目根目录下 /meta/ai-traces/ 存放 AI 辅助记录（审核日志、生成草稿）。

定期发布“自举报告”，统计 AI 贡献度、准确率、用户反馈，供社区审查。

第八编：异常处理与审计
第27条 异常响应
异常类型	处理逻辑
Skill 越权调用工具	工具层拦截，返回错误，记录审计，可配置为自动禁用该 Skill
守门员审查不确定	风险评分在阈值附近时，暂停安装，请求用户人工判断，并记录案例
模型调用失败	自动降级备用模型，若全失败则返回友好错误，提示稍后重试
隐私管道脱敏失败	拒绝处理该请求，记录日志，并通知用户增强脱敏规则
审计日志写入失败	触发内核警报，停止敏感操作，直到存储恢复
第28条 审计日志规范
日志包含：时间戳、工作区ID、调用方（Skill/Agent）、操作类型、参数摘要、结果、耗时。

日志独立存储（audit.db），只追加，定期轮转（如按月）。

用户可在设置中查看、导出、清空（清空前需确认）。

第九编：宪法修正
第29条 修正流程
任何对宪法的修改必须：

由用户或内核开发者提交提案。
经过安全影响评估（可由 AI 辅助）。
在项目社区公示至少 7 天。
获得用户（或代表委员会）批准。
修正案生效后，所有模块必须同步更新，并在文档中注明版本变更。

本宪法自 2026 年 3 月起生效，指导 eVoiceClaw Desktop v3 及其后续版本的所有设计与实现。任何组件——无论是人、AI 还是代码——都不得违背其条款。
