## Getting Started

```bash
git clone https://github.com/your-org/evoiceclaw-desktop-v3.git
cd evoiceclaw-desktop-v3

# Backend
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Frontend
cd ../frontend
npm install
```

Set up your local config:

```bash
cd backend
cp config.example.us.yaml config.yaml
cp secrets.yaml.example secrets.yaml
# Edit secrets.yaml — add at least one API key to run tests
```

## Developing a Skill

A **Skill** is a natural-language "program" described in a `SKILL.md` file. When installed, the Gatekeeper LLM reviews the Skill and generates `ACTIONS.yaml` — a machine-readable action allowlist that enforces the Skill's declared intent at runtime.

### Skill file format

```
data/skills/<skill_id>/
├── SKILL.md        # Human-readable description (your authorship)
├── ACTIONS.yaml    # Machine-generated by Gatekeeper (do not hand-edit)
└── VERSION.json    # Metadata (auto-generated)
```

### Writing SKILL.md

`SKILL.md` should describe:

1. **What the Skill does** — in plain language, one paragraph
2. **What data it accesses** — list external APIs, file paths, databases
3. **What actions it performs** — HTTP calls, shell commands, file writes
4. **What it does NOT do** — explicitly rule out sensitive operations

**Example:**

```markdown
# WeatherSkill

Queries real-time weather data for any city and formats the result
for display in the chat interface.

## Data Access
- External API: https://api.open-meteo.com/v1/forecast (GET only)
- No local file access
- No user personal data

## Actions
- HTTP GET requests to api.open-meteo.com
- Parse JSON response and format text summary

## Explicitly Out of Scope
- No POST/PUT/DELETE requests
- No access to user location beyond the city name provided in the query
- No data storage or caching
```

### Gatekeeper review

When you install a Skill (via UI or `POST /api/v1/skills/install`), the Gatekeeper LLM:

1. Reads your `SKILL.md`
2. Identifies all declared actions
3. Rewrites the content to remove or constrain unsafe actions
4. Generates `ACTIONS.yaml` with `command` and `pattern` entries

The Gatekeeper's `System Prompt` is private (stored in `gatekeeper_prompt.yaml`, never committed). This is by design — it follows the "AI governs AI" principle where the Gatekeeper is the sole legislative authority for Skill safety rules.

### Testing your Skill locally

```bash
# Install via API
curl -X POST http://localhost:28772/api/v1/skills/install \
  -H "Content-Type: application/json" \
  -d '{"skill_id": "weather", "content": "<paste SKILL.md content>"}'

# Check the generated ACTIONS.yaml
cat backend/data/skills/weather/ACTIONS.yaml

# Invoke in chat
# Ask the LLM: "What's the weather in Tokyo?"
```

## Hard Tool Development

**Hard tools** are Python functions registered in the `ToolRegistry`. Unlike Skills, they are compiled into the system and available to the LLM via Function Calling.

### Adding a new built-in tool

1. Create a file under `backend/app/kernel/tools/builtin/`:

```python
# backend/app/kernel/tools/builtin/my_tool.py
from app.kernel.tools.protocol import SkillProtocol, ToolResult

class MyTool(SkillProtocol):
    name = "my_tool"
    description = "One-sentence description for the LLM."
    required_permissions = ["my_permission"]
    security_level = "standard"   # or "elevated" / "restricted"

    parameters = {
        "type": "object",
        "properties": {
            "input": {"type": "string", "description": "..."}
        },
        "required": ["input"]
    }

    async def execute(self, input: str, **kwargs) -> ToolResult:
        # Implementation
        return ToolResult(success=True, data={"result": "..."})
```

2. Register it in `backend/app/main.py` inside the `lifespan` function:

```python
from app.kernel.tools.builtin.my_tool import MyTool
tool_registry.register(MyTool())
```

3. Write tests in `backend/tests/test_my_tool.py`.

### Tool security checklist

- [ ] Does the tool access the network? → Use `NetworkGuard.check_url()` before every request
- [ ] Does the tool write files? → Validate path is inside an allowed workspace directory
- [ ] Does the tool run shell commands? → Route through `ShellSandbox.execute_sandboxed()`
- [ ] Does the tool handle user data? → Ensure data flows through the Privacy Pipeline
- [ ] Does the tool need elevated permissions? → Set `security_level = "elevated"` and add an `ElevationRequest`

## Code Standards

### Python (backend)

- **Style**: PEP 8, enforced via `ruff` (planned)
- **Type hints**: Required for all public functions
- **Naming**: `snake_case` for variables/functions, `PascalCase` for classes
- **File length**: 200–800 lines per file; split into modules if longer
- **Async**: `async/await` throughout; no `time.sleep()` in async contexts
- **Error handling**: Only at system boundaries (user input, external APIs); do not catch-and-swallow internal errors
- **Secrets**: Never hard-code; read from `secrets.yaml` via `load_config()`
- **Comments**: Only where logic is non-obvious; no docstrings on trivial getters

```python
# Good — type hints, snake_case, async
async def check_url(url: str) -> tuple[bool, str]:
    ...

# Good — boundary validation only
async def http_request(url: str, method: str) -> ToolResult:
    allowed, reason = network_guard.check_url(url)
    if not allowed:
        return ToolResult(success=False, error=reason)
    ...
```

### TypeScript (frontend)

- **Framework**: React 19, functional components only
- **Naming**: `PascalCase` components, `camelCase` variables, `kebab-case` files
- **State**: Prefer hooks (`useState`, `useReducer`); no class components
- **API calls**: Through `src/lib/api.ts`; never `fetch()` directly in components
- **i18n**: All user-visible strings must use `useTranslation()` — no hardcoded Chinese or English in JSX

### Commit messages

Use imperative mood in Chinese (per project convention):

```
添加 WeatherSkill 示例和安装指南
修复 NetworkGuard 子域匹配误判
重构 SmartRouter 意图分类缓存逻辑
```

## Testing

### Backend

```bash
cd backend
python3 -m pytest tests/ -v           # all tests
python3 -m pytest tests/test_shell_sandbox.py -v   # single file
python3 -m pytest -k "whitelist" -v    # by keyword
```

Tests use `pytest-asyncio` in `auto` mode — mark async tests with `@pytest.mark.asyncio` or let auto-detection handle it.

**Coverage areas (280 tests):**

| File | Area |
|------|------|
| `test_shell_sandbox.py` | Layer 1/2/3 sandbox, env var filtering, ContextVar isolation |
| `test_network_guard.py` | Cloud metadata blocking, private IP, whitelist matching |
| `test_permission_broker.py` | Elevation requests, approve/deny/timeout |
| `test_rate_limiter.py` | Sliding window, 429 responses, exempt paths |
| `test_audit.py` | Log events, query filters, trace_id uniqueness |
| `test_privacy_pipeline.py` | Isolator, entity mapper, UUID recovery |
| `test_gatekeeper.py` | Prompt loading, JSON parsing, malicious Skill stripping |

When adding a feature, add tests. When fixing a bug, add a regression test first.

### Frontend

```bash
cd frontend
npm run test    # vitest (unit tests)
npm run build   # TypeScript type check + production build
```

## Pull Request Process

1. **Fork** the repository and create a branch: `feature/my-feature` or `fix/issue-123`
2. **Read** the files you are about to modify before editing
3. **Write tests** for your changes
4. **Run the full test suite** — all tests must pass
5. **Build the frontend** — `npm run build` must succeed
6. **Update docs** — if you add a file, update `docs/CODEBASE_INDEX.md`; if you complete a backlog item, update `docs/BACKLOG.md`
7. **Open a PR** with:
   - A clear description of *what* and *why*
   - Test results (paste `pytest` summary)
   - Any breaking changes or migration notes

### PR review criteria

- Does not introduce security regressions (network access, file write, shell execution without sandbox)
- All existing tests still pass
- New code follows the style and security checklist above
- Sensitive data is not logged or transmitted in plaintext

## Security Policy

### Reporting vulnerabilities

Please **do not** open a public GitHub issue for security vulnerabilities. Instead, email the maintainers directly (see `SECURITY.md` when available).

### What we consider a vulnerability

- Bypass of the Privacy Pipeline (sensitive data reaches LLM in plaintext)
- Bypass of NetworkGuard (outbound call to private IP or unlisted domain)
- Shell Sandbox escape (command execution outside the declared Skill scope)
- Gatekeeper bypass (malicious Skill installed without review)
- API key leakage (key appears in logs, SSE events, or API responses)

### What we do NOT consider a vulnerability

- A Skill that behaves unexpectedly if the user deliberately installs an untrusted Skill
- Rate limiting bypass via multiple client IPs (this is infrastructure-level, not application-level)

## Design Principles

When in doubt, follow these principles (from `docs/constitution.md`):

1. **Security first, features second** — every data flow must go through the privacy pipeline; every Skill must be reviewed by the Gatekeeper
2. **Minimum privilege** — a Skill may only use the tools it has declared
3. **Auditability** — every operation must produce an audit record with `trace_id`
4. **No secrets in code** — API keys, passwords, and tokens go in `secrets.yaml` only
5. **AI governs AI** — the Gatekeeper LLM is the sole authority for Skill safety rules; do not hard-code safety decisions in application logic
